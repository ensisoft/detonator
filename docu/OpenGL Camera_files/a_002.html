            YuriiB candy lau 10 months ago

            Short answer: It's because camera's coordinate system was constructed in a such way.

            Long answer: In order to construct camera coordinate system you need to have a look-at vector and one of the world's vectors. Lets say it would be UP vector, but could be any other. Now you can construct camera's X axis by cross-product look-at and UP vectors. Now you need to construct camera's Y axis. Following the same logic we need to cross-product camera's X axis vector and look-at vector. Now when you have X and Y camera's vectors you can following the same rule construct camera's Z vector which will point into opposite direction as your look-at vector. Why just no to use look-at vector for camera's Z? It's because as soon as you started calculating camera's axis you need to do for all direction without any exceptions. Just be consistent :)

                            songho Mod lilith a year ago edited

                            Suppose to rotate a virtual camera +30 degree along X-axis. It is equivalent to rotate the entire scene in opposite direction, -30 degree on X-axis. (This is what we actually do.)

                            This opposite rotation is acquired from finding the inverse matrix of the original rotation matrix.

                            The 3x3 matrix rotating only X-axis is defined here.

                            [ 1 0  0]
                            [ 0 C -S]
                            [ 0 S  C]

                            And its inverse matrix is

                            [ 1  0  0]
                            [ 0  C  S]
                            [ 0 -S  C]


                            You can verify it by inserting actual values (30 and -30) into the matrix

                            sin(30) = 0.5,  sin(-30) = -0.5
                            cos(30) = 0.87, cos(-30) = 0.87

The z/forward vector is the vector from the object to you (this threw me off a bit, but just note that the positive z/forward is the opposite of where you're looking, so that's why you need that the z base be from the object to you/eye, and not from you/eye to object). The new basis then are:
                                                                                                                          x y z
                                                                                                                          0 0 -1 3
                                                                                                                          0 1 0 x 0
                                                                                                                          1 0 0 0

                                                                                                                          this gives (0,0,3) meaning, it turns the world to the right - but leaves you in place; or, it gives you the new coordination, but based on the original coordinates. What you actually want to do is the opposite. I.e. you want to turn the world to the left, to fake as if you turned right. The inverse matrix is
                                                                                                                          0 0 1
                                                                                                                          0 1 0
                                                                                                                          -1 0 0

                                                                                                                          multiplying by (3,0,0) will give (0,0,-3) - now we can see the object as it is just 3 units infront of us (again, we are looking towards the negative z axis); or, it gives you the new coordination based on the new coordinates.

                                                                                                                                                > Practically why do we need this three things to calculate the rotation matrix?

                                                                                                                                                One question might be why would the x coordinate of a rotated vector be dependent on y and z starting values?

                                                                                                                                                If you study the diagrams on this page carefully, you will hopefully see that it makes sense http://www.songho.ca/opengl... .

                                                                                                                                                But for a little intuition think of a line that is the axis of rotation (any line in space) and think of a point not on the line but close to the line being rotated 1 radian (or whatever angle) around the line. It is now at (x',y',z'), where those new coordinates are not far from the starting (x,y,z). Now imagine if that point had instead a much much larger starting y coordinate. Likely the point would be much further away from the line, so after now doing 1 radian rotation the new ending pt would probably also be much further away from the line as before. And both points would be far from each other. Likely the x' value would be much different and larger in magnitude. So changing the y value can definitely have an effect on the x' value. Basically, the final coordinate values x', y', and z' each depend on each of the starting values, x, y, z.. in general.

                                                                                                                                                OK, to see why songho uses those 3 vectors to compose the rotation matrix (but which will need to be inversed) go and reread that link above. It addresses that, especially at the very top. Basically if you just multiply a rotation matrix and the "left" vector, you will get the first column of the matrix as the final rotated value. (left vector is 1 0 0). Just multiply it out yourself to verify. So the rotation matrix first column is the new left value after rotation. Keep going to see that the second column of the rotation matrix is the new value of the up vector. The third column is the new forward coordinates. So that is how we can create the rotation matrix. The three columns are (in that order) the new coordinates of where the left up and forward vectors will end up.

                                                                                                                                                Can we have confidence that this "rotation" matrix that rotates left up and forward to the new values will also rotate any vector the same way? Yes. Let's see.

                                                                                                                                                If the vector is (5 0 0), it will rotate to where the (1 0 0) went but with each new coordinate 5 times larger. (1 0 0) -> (a b c) means (5 0 0) -> (5a 5b 5c). We can repeat this. (0 1 0 ) -> (d e f) means (0 3 0) -> (3d 3e 3f). And .. (0 0 6) -> (6g 6h 6i).

                                                                                                                                                Since vector (5 3 6) is just the sum of the above, it's new rotated position will be the sum of the other 3 rotated positions (think rigid motion).

                                                                                                                                                (5 3 6) = (5 0 0) + (0 3 0) + (0 0 6) ->
                                                                                                                                                (5a 5b 5c) + (3d 3e 3f) + (6g 6h 6i) =
                                                                                                                                                (5a+3d+6g 5b+3e+6h 5c+3f+6i).

                                                                                                                                                And this vector is(*) just the "rotational" matrix multiplied with the (5 3 6). So multiplying a generic vector by the "rotation" matrix gave the same value as we would expect to be the rotated point as seen earlier by adding up all the rotated pieces. The formalism skipped is to explain more mathematically what a rigid motion is, but intuitively and in pictures we can see it.

                                                                                                                                                [b e h]

                                                                                                                                                [5c+3f+6i]
